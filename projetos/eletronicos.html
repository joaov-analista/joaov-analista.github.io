Contexto do Projeto

O projeto nasceu de um enigma real de negócio: o faturamento da loja caía consistentemente em abril — mas o Ticket Médio e os Itens por Pedido continuavam saudáveis. A contradição indicava que o problema não estava no comportamento do cliente nem no mix de produtos.

A abordagem foi estruturada em três etapas. Na primeira, realizei um ETL rigoroso no MySQL: converti strings para formatos nativos de DATE com STR_TO_DATE e para DECIMAL eliminando símbolos de moeda, executei testes de frequência com LIKE para identificar registros corrompidos e implementei o StoreKey 0 como "Loja Digital" para tratar vendas sem identificação de loja física — evitando perda de dados no modelo dimensional.

Na segunda etapa, construí o dashboard no Power BI com KPIs de Ticket Médio, Itens por Pedido (IPP) e Margem de Contribuição, além de medidas DAX para separar o faturamento de Lojas Físicas vs. E-commerce. Usei a Árvore de Decomposição para realizar o drill-down automático dos fatores que mais impactavam a queda de abril. Na terceira etapa, projetei o faturamento de abril para um cenário de 30 dias de operação plena — mensurando exatamente quanto a empresa deixou de ganhar com os fechamentos.

Stack utilizada

MySQL — ETL com STR_TO_DATE e DECIMAL

-- Etapa 1: ETL e Tratamento de Dados | MySQL
-- Autor: João Vítor de Freitas
-- ================================================

-- 1. Conversão de datas: string → DATE nativo
UPDATE vendas
SET data_venda = STR_TO_DATE(data_venda_str, '%d/%m/%Y')
WHERE data_venda IS NULL;

-- 2. Limpeza de valores monetários: string → DECIMAL
UPDATE vendas
SET faturamento = CAST(
  REPLACE(REPLACE(faturamento_str, 'R$', ''), ',', '.') AS DECIMAL(12,2)
);

-- 3. Testes de frequência: identificar registros corrompidos
SELECT faturamento_str, COUNT(*) AS qtd
FROM vendas
WHERE faturamento_str LIKE '%[^0-9.,]%'  -- caracteres inesperados
GROUP BY faturamento_str
ORDER BY qtd DESC;

-- 4. Mapeamento de nulos por dimensão
SELECT
  COUNT(*) - COUNT(id_loja)     AS nulos_loja,
  COUNT(*) - COUNT(id_produto)  AS nulos_produto,
  COUNT(*) - COUNT(id_cliente)  AS nulos_cliente,
  COUNT(*) - COUNT(data_venda)  AS nulos_data
FROM vendas;

-- 5. Tratar vendas sem loja física → StoreKey 0 (Loja Digital)
UPDATE vendas
SET id_loja = 0
WHERE id_loja IS NULL;

INSERT IGNORE INTO dim_loja (id_loja, nome_loja, canal)
VALUES (0, 'Loja Digital', 'E-commerce');

-- 6. Dias de operação por mês (base para o Efeito Calendário)
SELECT
  YEAR(data_venda)                     AS ano,
  MONTH(data_venda)                    AS mes,
  COUNT(DISTINCT data_venda)           AS dias_abertos,
  ROUND(SUM(faturamento), 2)           AS faturamento_total,
  ROUND(SUM(faturamento)
        / COUNT(DISTINCT data_venda), 2) AS fat_medio_diario
FROM vendas
GROUP BY ano, mes
ORDER BY ano, mes;

-- 7. Revenue Gap: projeção abril para 30 dias de operação plena
WITH abril AS (
  SELECT
    COUNT(DISTINCT data_venda)           AS dias_abertos,
    ROUND(SUM(faturamento), 2)           AS fat_real,
    ROUND(SUM(faturamento)
          / COUNT(DISTINCT data_venda), 2) AS fat_diario
  FROM vendas
  WHERE MONTH(data_venda) = 4
)
SELECT
  dias_abertos,
  fat_real                                       AS faturamento_realizado,
  ROUND(fat_diario * 30, 2)                      AS faturamento_projetado_30d,
  ROUND(fat_diario * 30 - fat_real, 2)           AS revenue_gap
FROM abril;

# REVENUE GAP — LOJA DE ELETRÔNICOS
# Etapa 2 & 3: BI e Análise Preditiva | Python
# Autor: João Vítor de Freitas
# ================================================

import pandas as pd
import matplotlib.pyplot as plt
import matplotlib.ticker as mtick
import seaborn as sns
import numpy as np
from scipy import stats

sns.set_theme(style='whitegrid')
plt.rcParams['figure.dpi'] = 120

# ── 1. Carregamento dos dados ───────────────────
df = pd.read_csv('vendas_eletronicos.csv', parse_dates=['data_venda'])
df['mes']      = df['data_venda'].dt.month
df['ano']      = df['data_venda'].dt.year
df['mes_nome'] = df['data_venda'].dt.strftime('%b')

# ── 2. Dias abertos e faturamento por mês ───────
mensal = (
    df.groupby('mes')
      .agg(
          dias_abertos=('data_venda', 'nunique'),
          faturamento=('faturamento', 'sum'),
          margem=('margem_contribuicao', 'mean')
      )
      .reset_index()
)
mensal['fat_medio_diario'] = (
    mensal['faturamento'] / mensal['dias_abertos']
)
print(mensal[['mes','dias_abertos','faturamento','fat_medio_diario']].round(2))

# ── 3. Correlação: Dias Abertos × Faturamento ───
r, p = stats.pearsonr(mensal['dias_abertos'], mensal['faturamento'])
print(f"\nCorrelação de Pearson (r={r:.3f}, p={p:.4f})")

fig, ax = plt.subplots(figsize=(8, 5))
ax.scatter(mensal['dias_abertos'], mensal['faturamento'],
           color='#2563EB', s=80, zorder=5)
# destaque para abril
abril = mensal[mensal['mes'] == 4]
ax.scatter(abril['dias_abertos'], abril['faturamento'],
           color='#EF4444', s=140, zorder=6, label='Abril')
m, b = np.polyfit(mensal['dias_abertos'], mensal['faturamento'], 1)
x_ln = np.linspace(mensal['dias_abertos'].min(),
                   mensal['dias_abertos'].max(), 100)
ax.plot(x_ln, m * x_ln + b, '--', color='#9CA3AF', linewidth=1.5)
ax.set_title(f'Dias Abertos × Faturamento (r={r:.2f})',
             fontsize=13, fontweight='bold')
ax.set_xlabel('Dias de Operação no Mês')
ax.yaxis.set_major_formatter(
    mtick.FuncFormatter(lambda x, _: f'R$ {x/1000:.0f}K'))
ax.legend()
plt.tight_layout()
plt.savefig('outputs/correlacao_dias_faturamento.png', bbox_inches='tight')
plt.show()

# ── 4. Revenue Gap: projeção de abril para 30 dias ─
fat_diario_abril = abril['fat_medio_diario'].values[0]
fat_real_abril   = abril['faturamento'].values[0]
fat_proj_30d     = fat_diario_abril * 30
revenue_gap      = fat_proj_30d - fat_real_abril

print(f"\n── Revenue Gap (Abril) ──────────────────")
print(f"Faturamento real:        R$ {fat_real_abril:,.0f}")
print(f"Faturamento projetado:   R$ {fat_proj_30d:,.0f}")
print(f"Revenue Gap:             R$ {revenue_gap:,.0f}")
print(f"Margem de contribuição:  {abril['margem'].values[0]:.0%}")

Insights do Projeto

O Efeito Calendário: o problema era operacional, não comercial
A queda de faturamento em abril não tinha origem comercial. Através de um gráfico de dispersão, provei a correlação direta entre Dias Abertos e Faturamento — abril simplesmente tinha menos dias de operação. O comportamento do cliente e o mix de produtos estavam saudáveis.
Abril era o mês mais produtivo — com 60% de margem de contribuição
A métrica de Faturamento Médio Diário revelou que, enquanto estava aberta, a loja era mais produtiva em abril do que em outros meses. A margem de contribuição de 60% neste mês era superior à média — ou seja, abril era uma oportunidade disfarçada de problema.
Revenue Gap: queda virou oportunidade de expansão operacional
Projetando o faturamento de abril para 30 dias de operação plena, foi possível mensurar exatamente quanto a empresa deixou de ganhar com os fechamentos — transformando uma queda de faturamento em argumento concreto para revisão do calendário operacional da loja.
